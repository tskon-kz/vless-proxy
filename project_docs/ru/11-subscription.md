# Подписки (`core/subscription.py`)

[English](../en/11-subscription.md)

Подписки — единственный источник прокси-серверов. Подписка — это URL, возвращающий список `vless://` ссылок в виде plain text или base64.

## Настройка

```env
SUBSCRIPTION_URLS=["https://sub.example.com/token"]
SUBSCRIPTION_FETCH_INTERVAL=1800   # секунды, по умолчанию 30 мин
SUBSCRIPTION_TIMEOUT=30            # таймаут загрузки, сек
```

Несколько подписок:
```env
SUBSCRIPTION_URLS=["https://sub1.example.com/token","https://sub2.example.com/token"]
```

## Процесс загрузки

1. `SubscriptionFetcher.fetch(url)` — HTTP GET с `User-Agent: v2rayN/6.0`.
2. Тело ответа пробуется как base64; если после декодирования содержит `vless://` — используется декодированный текст, иначе — исходный.
3. Все строки с `vless://` передаются в `parse_vless_list()`.
4. Вызывается `storage.replace_subscription_proxies()`:
   - Новые URI → вставляются со статусом `pending`
   - Существующие URI → статус не меняется (мёртвые сбрасываются в `pending`)
   - URI, исчезнувшие из подписки → помечаются `dead`
5. Немедленно запускается цикл проверок для новых/сброшенных прокси.

## Поллинг

Каждая подписка работает в отдельной asyncio-задаче (`_start_poller`):
- Если `last_fetch = NULL` (старт или первый запуск) → загрузить немедленно.
- Если недавно загружалась → ждать оставшийся интервал.
- После каждой загрузки — sleep на `SUBSCRIPTION_FETCH_INTERVAL` секунд.

## Дедупликация URI

`raw_uri` (ключ идентификации в БД) строится из URI с query-параметрами, отсортированными по алфавиту, и удалённым фрагментом. Это гарантирует, что один и тот же сервер, возвращённый с разным порядком параметров, распознаётся как тот же прокси и не создаёт дубликатов.

## `SubscriptionManager`

| Метод | Описание |
|---|---|
| `startup()` | Зарегистрировать URL из конфига в БД, запустить задачи-поллеры |
| `refresh(sub_id)` | Загрузить и применить одну подписку немедленно |
| `refresh_all()` | Загрузить все подписки немедленно |
| `shutdown()` | Отменить все задачи-поллеры |
